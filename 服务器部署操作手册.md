## 1. 部署架构说明

### 为什么推荐 VPS，而不是 Vercel

调用 `/api/generate` 等生成类接口时，若 Suno 要求验证码，服务会：

1. 检测是否需要 CAPTCHA
2. 启动 **Playwright + Chromium** 打开 `suno.com/create`
3. 自动触发 hCaptcha，调用 **2Captcha** 求解
4. 拿到 token 后再请求 Suno 官方生成 API

Vercel Serverless 环境 **无法稳定运行完整 Chromium**（体积、超时、无持久磁盘等限制），一旦触发验证码就会报 `Executable doesn't exist` 或超时。**生产环境请使用 VPS 或 Docker。**

### 服务端口

项目默认监听 **3021** 端口（见 `package.json` 中 `dev` / `start` 脚本）。对外访问通常通过 Nginx 反代到 80/443。

## 3. 首次部署步骤

以下在 VPS 上以普通用户（如 `ubuntu`）执行，**全程不要用 `sudo npm install`**，否则浏览器会装到 root 目录，运行时找不到。

### 3.1 安装 Node.js

```bash
# 示例：使用 NodeSource 安装 Node 20
curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
sudo apt-get install -y nodejs
node -v && npm -v
```

### 3.2 拉取代码并安装依赖

```bash
cd ~
git clone https://github.com/gcui-art/suno-api.git   # 或你的 fork / 私有仓库
cd suno-api

cp .env.example .env
nano .env   # 填写 SUNO_COOKIE、TWOCAPTCHA_KEY 等，见下文

# 严格按 lock 安装，避免 Playwright 版本漂移
npm ci
```

### 3.3 安装 Playwright 浏览器（关键）

```bash
# 只安装 headless-shell（项目 BROWSER_HEADLESS=true 时足够）
npm run playwright:install

# 安装 Ubuntu 系统依赖（必须）
sudo npm run playwright:install-deps
```

验证浏览器文件存在（revision 号须与项目一致，当前为 **1169**，见 [第 5 节](#5-playwright-与-chromium-版本问题)）：

```bash
ls -la ~/.cache/ms-playwright/chromium_headless_shell-1169/chrome-linux/headless_shell
node -e "console.log(require('rebrowser-playwright-core/package.json').version)"
# 应输出 1.52.0
```

若 CLI 安装卡住或下载失败，使用手动脚本：

```bash
chmod +x scripts/install-chromium-headless-linux.sh
bash scripts/install-chromium-headless-linux.sh
```

### 3.4 构建并启动

```bash
npm run build

# 取消 DISPLAY，避免 SSH X11 转发导致浏览器弹窗卡住
unset DISPLAY
npm run start
```

验证：

```bash
curl http://127.0.0.1:3021/api/get_limit
```

返回 `credits_left`、`monthly_limit` 等字段即表示 API 与 Cookie 基本正常。

### 3.5 使用 PM2 守护进程（推荐）

```bash
sudo npm install -g pm2

cd ~/suno-api
pm2 start npm --name suno-api -- run start
pm2 save
pm2 startup   # 按提示执行生成的命令，实现开机自启
```

PM2 环境变量：可在项目根目录放 `.env`，或在 `ecosystem.config.js` 中配置；确保包含 `BROWSER_HEADLESS=true`。

### 3.6 Nginx 反向代理（示例）

```nginx
server {
    listen 80;
    server_name sunoapi.example.com;

    location / {
        proxy_pass http://127.0.0.1:3021;
        proxy_http_version 1.1;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        # 生成接口可能较慢，适当加大超时
        proxy_read_timeout 300s;
        proxy_connect_timeout 60s;
    }
}
```

HTTPS 可使用 Certbot 申请证书。

## 4. 环境变量说明

复制 `.env.example` 为 `.env` 后，至少配置以下项：

```bash
# 必填：从浏览器登录 suno.com 后复制 Cookie（见 README_CN.md）
SUNO_COOKIE=

# 必填：过 hCaptcha 时使用，在 https://2captcha.com 注册获取
TWOCAPTCHA_KEY=

# 浏览器（VPS 推荐如下组合）
BROWSER=chromium
BROWSER_HEADLESS=true
BROWSER_DISABLE_GPU=true
BROWSER_LOCALE=en
BROWSER_GHOST_CURSOR=false

# 验证码失败时在 /tmp 保存截图，便于排查
CAPTCHA_DEBUG_SCREENSHOT=false

# 可选：API Bearer 鉴权
API_AUTH_ENABLED=false
API_SECRET=
```

### 获取 SUNO_COOKIE

1. 浏览器登录 [suno.com/create](https://suno.com/create)
2. F12 → Network → 刷新页面
3. 找到含 `client?_clerk_js_version` 的请求
4. 复制 Request Headers 中的完整 `Cookie` 值

Cookie 过期或失效时，浏览器会跳转到登录页，日志会出现 `Browser redirected to sign-in`，需重新复制 Cookie。

### 启动前务必 unset DISPLAY

通过 Xshell 等 SSH 客户端连接时，若开启了 X11 转发且 `BROWSER_HEADLESS` 未生效，Chromium 会尝试在本地弹出窗口，导致请求长时间挂起。启动前执行：

```bash
unset DISPLAY
```

Xshell 中也可在「会话属性 → SSH → X11 转发」关闭该功能。

## 5. Playwright 与 Chromium 版本问题

这是 VPS 部署中最常见的一类错误。

### 5.1 现象

日志中出现类似：

```text
CAPTCHA required. Launching browser...
Executable doesn't exist at .../ms-playwright/chromium_headless_shell-1169/.../headless_shell
Please run: npx playwright install
```

或：已安装浏览器，但 revision 目录名与运行时要求的不一致（例如缓存里是 `1228`，程序要找 `1169`）。

### 5.2 根本原因

本项目使用的是 **`rebrowser-playwright-core`**（反检测补丁版 Playwright），**不是** `@playwright/test` 或全局的 `playwright` 包。

浏览器 revision（构建号）与 npm 包版本 **一一对应，不能混用**：

| Chromium headless-shell revision | 对应 npm 包 | 说明 |
|----------------------------------|-------------|------|
| **1169** | `rebrowser-playwright-core@1.52.0` | **当前项目锁定版本** |
| 1148 | `rebrowser-playwright-core@1.49.1` | 旧版 lock |
| 1228 | `playwright-core@1.61.0` | 若执行了 `npx playwright install` 会装这个 |

常见误操作：

| 错误命令 | 后果 |
|----------|------|
| `npx playwright install` | 安装 1.61.0 的浏览器（1228），与项目不匹配 |
| `npx playwright-core install chromium` | 只装普通 Chromium，**未装 headless-shell** |
| `sudo npm install` | 浏览器装到 root 的 `~/.cache`，当前用户找不到 |
| 不用 `npm ci`、lock 漂移 | node_modules 版本与 package-lock 不一致，revision 对不上 |

### 5.3 正确安装方式

**始终使用项目内置 CLI：**

```bash
cd ~/suno-api
npm ci

# 查看实际版本
node -e "console.log(require('rebrowser-playwright-core/package.json').version)"
# 期望：1.52.0

# 安装 headless-shell（headless 模式必需）
npm run playwright:install
# 等价于：
# node node_modules/rebrowser-playwright-core/cli.js install chromium-headless-shell

sudo npm run playwright:install-deps

# 验证 1169 目录
ls ~/.cache/ms-playwright/chromium_headless_shell-1169/chrome-linux/headless_shell
```

### 5.4 只装 headless-shell，不必装完整 Chromium

`.env` 中 `BROWSER_HEADLESS=true` 时，Playwright 启动的是 **`chromium_headless_shell`**，不是完整 `chromium`。  
只需安装 `chromium-headless-shell` 即可，体积更小、安装更快。

### 5.5 CLI 下载卡在 100%

`node .../cli.js install` 下载完成后长时间无输出，通常是在 **解压大 zip**（小内存 VPS 可能 2–5 分钟，或 OOM 卡死）。

处理：

```bash
# 1. Ctrl+C 终止卡住的进程
# 2. 检查资源
df -h ~
free -h

# 3. 使用手动安装脚本
bash scripts/install-chromium-headless-linux.sh
```

脚本默认安装 revision **1169**，可通过环境变量覆盖：

```bash
PLAYWRIGHT_CHROMIUM_REVISION=1169 bash scripts/install-chromium-headless-linux.sh
```

安装成功后可选写入 `.env`：

```bash
PLAYWRIGHT_CHROMIUM_EXECUTABLE_PATH=/home/ubuntu/.cache/ms-playwright/chromium_headless_shell-1169/chrome-linux/headless_shell
```

### 5.6 缓存里有错误 revision 时

若曾误装 1228，可删除错误目录后重装：

```bash
rm -rf ~/.cache/ms-playwright/chromium_headless_shell-1228
rm -rf ~/.cache/ms-playwright/chromium-1228

npm run playwright:install
```

**不要**再执行 `npx playwright install` 或 `playwright-core install`（不带 rebrowser 路径）。

### 5.7 升级项目后 revision 可能变化

若将来升级 `rebrowser-playwright-core`，revision 可能从 1169 变为其他数字。升级后务必：

```bash
git pull
rm -rf node_modules
npm ci
npm run playwright:install
sudo npm run playwright:install-deps
npm run build
```

并核对 `node_modules/rebrowser-playwright-core/browsers.json` 中的 `chromium-headless-shell.revision`。

## 6. Suno 模型与账户权限对照

生成接口（`/api/generate`、`/api/custom_generate`、`/api/extend_audio` 等）通过 `model` 字段映射为 Suno API 的 `mv` 参数。  
**模型必须与 Suno 账户套餐匹配**，否则会返回：

```json
{
  "status_code": 403,
  "detail": "You don't have access to this model. Upgrade your plan to use it.",
  "error_type": "permission_denied"
}
```

这与 Cookie、验证码无关；验证码已通过仍 403，几乎都是 **model 选错**。

### 6.1 网页显示名与 API 值对照

| Suno 网页显示 | API `model` 值 | Free Plan | Pro / Premier |
|---------------|----------------|-----------|---------------|
| **v4.5-all**（Best free model） | **`chirp-auk-turbo`** | ✅ 可用 | ✅ |
| v4.5 Pro | `chirp-auk` | ❌ 403 | ✅ |
| v4.5+ Pro | `chirp-bluejay` | ❌ 403 | ✅ |
| v5 Pro | `chirp-crow` | ❌ 403 | ✅ |
| v5.5 Pro | `chirp-fenix` | ❌ 403 | ✅ |
| v4 Pro | `chirp-v4` | ❌ 403 | ✅ |
| ~~v3.5~~ | ~~`chirp-v3-5`~~ | ❌ 已下线 | ❌ |
| ~~v3.0~~ | ~~`chirp-v3-0`~~ | ❌ 已下线 | ❌ |

### 6.2 易错点

1. **`chirp-auk` ≠ `chirp-auk-turbo`**  
   Free 账号网页默认是 **v4.5-all**，对应 API 值是 **`chirp-auk-turbo`**，不是 `chirp-auk`（Pro 模型）。

2. **服务端默认模型**  
   代码中 `DEFAULT_MODEL = 'chirp-auk-turbo'`（见 `src/lib/SunoApi.ts`）。请求体 **省略 `model` 字段** 时使用该默认值。

3. **显式传错 model 会覆盖默认值**  
   即使服务端默认正确，若 Postman / 客户端仍传 `"model": "chirp-v3-5"` 或 `"chirp-auk"`，Free 账号仍会 403。

### 6.3 Free Plan 正确请求示例

```json
POST /api/generate
Content-Type: application/json

{
  "prompt": "A high-energy dance hall track",
  "make_instrumental": false,
  "model": "chirp-auk-turbo",
  "wait_audio": false
}
```

### 6.4 如何确认账号可用模型

1. 登录 [suno.com/create](https://suno.com/create)，查看模型下拉列表中带 **Pro** 标记的项。  
2. 调用 `/api/get_limit` 确认 Cookie 有效、账号有剩余积分。  
3. Pro 用户可在网页上选择对应 Pro 模型后，API 中使用上表中的 `chirp-auk`、`chirp-crow` 等值。

## 7. 验证码（hCaptcha）流程与排错

### 7.1 何时触发

Suno 在部分请求前要求 CAPTCHA。服务会先调用 Suno 接口检测；若返回 `required: true`，则进入浏览器自动化流程。

典型日志顺序：

```text
CAPTCHA required. Launching browser...
Waiting for Suno interface to load
Triggering the CAPTCHA
hCaptcha challenge is visible
Sending the CAPTCHA to 2Captcha
hCaptcha token received. Closing browser
```

之后才请求 `POST /api/generate/v2/`。

### 7.2 自动化流程概要

1. 启动 headless Chromium，注入 `SUNO_COOKIE`（含 `auth.suno.com` 域）
2. 打开 `https://suno.com/create`
3. 切换到 Simple 模式，在 **Song Description** 输入框填词
4. 点击 **Create song**（未填词时按钮为 disabled）
5. 等待 hCaptcha 挑战 iframe 出现
6. 截图发送至 2Captcha，自动点击/拖拽完成验证
7. 拦截生成请求，提取 token，关闭浏览器

### 7.3 Suno 新版 UI 与 hCaptcha 域名（2026）

Suno 改版后，旧选择器会失效。当前实现已适配：

| 元素 | 说明 |
|------|------|
| 输入框 | Song Description，`placeholder` 含 `Mellow folk metal` / `song about` 等 |
| 按钮 | `aria-label="Create song"`，不是泛匹配的 `Create` 或 `Create New Workspace` |
| hCaptcha iframe | `title="hCaptcha challenge"`，`src` 含 `frame=challenge` |
| hCaptcha 资源 | Suno **自建代理**：`hcaptcha-assets-prod.suno.com`、`hcaptcha-imgs-prod.suno.com` 等，**不仅是** `hcaptcha.com` |

若 Suno 再次改版，可能出现 `Could not find song description input` 或 `hCaptcha did not appear`，需更新 `src/lib/SunoApi.ts` 中的选择器。

### 7.4 必备配置

| 变量 | 作用 |
|------|------|
| `TWOCAPTCHA_KEY` | 2Captcha 账户密钥，未设置则无法求解 |
| `BROWSER_HEADLESS=true` | VPS 无图形界面，必须 headless |
| `SUNO_COOKIE` | 须为登录态；失效会跳转 sign-in 页 |
| `CAPTCHA_DEBUG_SCREENSHOT=true` | 失败时在 `/tmp/suno-captcha-*.png` 保存截图 |

2Captcha 按次计费，验证码题型含点击、拖拽等，拖拽类会使用 `public/drag-instructions.jpg` 辅助识别。

### 7.5 常见失败场景

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| `Executable doesn't exist ... headless_shell` | 未装或 revision 不匹配 | 见 [第 5 节](#5-playwright-与-chromium-版本问题) |
| Xshell 弹出 X11 / 请求一直挂起 | 非 headless 或 `DISPLAY` 未 unset | `BROWSER_HEADLESS=true`，`unset DISPLAY` |
| `Browser redirected to sign-in` | Cookie 过期 | 重新复制 `SUNO_COOKIE` |
| `waiting for locator('.custom-textarea')` | 代码过旧 | 拉取最新代码并 `npm run build` |
| `hCaptcha did not appear after clicking Create song` | 点错按钮、Cookie 无效、页面改版 | 开启 `CAPTCHA_DEBUG_SCREENSHOT`，查看 `/tmp` 截图 |
| `Sending the CAPTCHA to 2Captcha` 后失败 | 密钥无效、余额不足、2Captcha 超时 | 检查 2Captcha 控制台 |
| 验证码通过但 403 | 模型权限 | 见 [第 6 节](#6-suno-模型与账户权限对照)，改用 `chirp-auk-turbo` |

### 7.6 调试命令

```bash
# 查看验证码调试截图
ls -la /tmp/suno-captcha-*.png

# 确认环境
grep -E 'BROWSER|TWOCAPTCHA|SUNO_COOKIE|CAPTCHA' .env
echo "DISPLAY=$DISPLAY"

# 前台运行观察完整日志
unset DISPLAY
npm run start
```

## 8. 日常运维与更新

### 8.1 更新代码

```bash
cd ~/suno-api
git pull
npm ci
npm run playwright:install      # 若 package-lock 中 rebrowser 版本有变
sudo npm run playwright:install-deps
npm run build
pm2 restart suno-api            # 或手动 unset DISPLAY && npm run start
```

### 8.2 Cookie 轮换

Suno Cookie 会过期。建议：

- 定期调用 `/api/get_limit` 监控
- 生成失败且日志含 `sign-in` 时更新 `SUNO_COOKIE` 并重启服务

### 8.3 日志

使用 PM2 时：

```bash
pm2 logs suno-api
pm2 logs suno-api --lines 200
```

### 8.4 文档与返回示例

`tests/*-response.json` 存放各 API 实际返回报文（命名规则：`{api路径尾部}-response.json`）。  
构建时 `scripts/sync-swagger-examples.mjs` 会将其同步到 Swagger 文档。

## 9. 常见问题速查

| 问题 | 解决方案 |
|------|----------|
| `next: Permission denied` | `chmod +x node_modules/.bin/*` 或 `rm -rf node_modules && npm ci` |
| `Could not find a production build in the '.next' directory` | 先执行 `npm run build` |
| `Can't resolve 'electron'`（构建失败） | 确保使用仓库内 `next.config.mjs`（已 alias `electron: false`） |
| `npm run dev` 不适合生产 | 使用 `npm run build` + `npm run start` 或 PM2 |
| Vercel 部署生成失败 | 改用 VPS；Vercel 无法稳定跑 Playwright |
| Free 账号 403 | 使用 `"model": "chirp-auk-turbo"`，不要用 `chirp-auk` / `chirp-v3-5` |
| 浏览器 revision 1169 vs 1228 | 只用 `npm run playwright:install`（rebrowser CLI），勿用 `npx playwright install` |
| 安装 Chromium 卡住 | 使用 `scripts/install-chromium-headless-linux.sh` |
| SSH 下浏览器弹窗 | `BROWSER_HEADLESS=true` + `unset DISPLAY` |

## 附录：Docker 部署（可选）

若希望与官方 Dockerfile 一致，可使用：

```bash
cp .env.example .env
# 编辑 .env
docker compose build && docker compose up -d
```

Docker 镜像内会执行 `npx playwright install chromium`，并设置 `BROWSER_DISABLE_GPU=true`。  
裸机部署更便于排查 Playwright 版本问题，本文档以裸机流程为主。
